## Motivation

The current `agent-harness` report is technically correct but hard to read. It exposes the filesystem and raw protocol better than it explains the run.

Terje's read: this should feel closer to the `pm` dashboard — overview first, drill-down second.

## Problem

Current report shape:

- run status and goal;
- sprint card;
- artifact path list;
- event history;
- raw JSON available in folders.

That proves the run happened, but it does not answer the useful questions quickly:

- What was the run trying to prove?
- What did planner decide?
- What did generator actually do or propose?
- What did evaluator attack?
- What changed in the code/design?
- What failed, pivoted, or needs review?
- Where do I click to inspect evidence?

## Desired shape

Build a PM-dashboard-like run surface:

```text
Run overview
├─ verdict / status / duration / branch / issue IDs
├─ goal in human language
├─ sprint cards with PASS/PIVOT/REJECT badges
├─ timeline: planner → generator → evaluator → verdict
├─ artifacts grouped by role and kind
├─ contract clauses and whether each passed
├─ changed files / proposed files
├─ evaluator issues and taste warnings
└─ deep-dive drawers for raw input/output/stdout/stderr
```

## Acceptance criteria

- Harness report opens as a readable dashboard, not a document dump.
- Top screen answers: goal, verdict, sprints, artifacts, next action.
- Role sections are visually distinct: planner, generator, evaluator.
- Artifact links are clickable and grouped by kind.
- Raw IO is still accessible but hidden behind details/disclosure controls.
- The report can be generated from the existing `.harness/runs/RUN-*` structure without requiring a server.
- Design should be portable to `agent-harness`, not hardcoded to Lifelines.

## Non-goals

- No live dashboard server yet.
- No React app yet unless static HTML becomes painful.
- No model/provider assumptions.
- No replacement for `pm`; this is a run report, not a planning database.
