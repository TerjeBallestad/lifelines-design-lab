Create a lightweight adversarial implementation loop inside the design lab, using `pm` issues as the work queue and the current seven design slices as the first test corpus.

Intent:
- Battle-test the planner/generator/evaluator workflow in a cheaper React/Vite design-lab surface before migrating the useful parts into the `pm` tool.
- Preserve the adversarial pressure of the Tycoon harness without dragging in Godot/headless tournament complexity too early.
- Make the process produce design decisions, not just generated code.

Proposed loop:
1. Planner selects one open `pm` issue and writes a tiny sprint contract: goal, touch surface, non-goals, expected player decision, verification gates.
2. Evaluator attacks the contract before implementation: asks what would make the slice fake, dashboard-only, unreadable, or not Lifelines-shaped.
3. Generator implements the agreed slice in the design lab.
4. Evaluator reviews the result against the contract, tests, browser behavior, and Lifelines design pressure.
5. Rook writes a compact verdict: PASS / REVISE / DISCARD, with what the slice taught.

Acceptance criteria:
- There is a documented design-lab harness workflow in the repo.
- The workflow uses `pm` issue IDs as inputs, not GitHub issues.
- It defines planner, generator, evaluator, and final-gate responsibilities.
- It defines contract artifacts and where they live.
- It includes a first dry-run against one existing design issue, preferably SB-002 or SB-008.
- It records what should later migrate into `pm` itself.

Non-goals:
- Do not port the full Tycoon harness.
- Do not require Claude/Anthropic as the default agent path.
- Do not build a dashboard before proving the loop.
- Do not make the process so heavy that it kills the design-lab speed.
